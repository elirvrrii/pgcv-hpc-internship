Git Installation & Global Configuration
Day 1 · ~30 mins · Ubuntu Server & local machine
0 / 7 ▼
Best practice: Always configure Git with your real name and institutional email before making any commits. This ensures proper attribution in the team repository.
01Install Git on Ubuntu: sudo apt install git -y
02Verify installation: git --version — should show git version 2.x.x
03Set your name globally: git config --global user.name "Your Full Name"
04Set your email: git config --global user.email "you@up.edu.ph"
05Set default branch name to main: git config --global init.defaultBranch main
06Set preferred editor to nano: git config --global core.editor nano
07Verify all settings: git config --global --list

🔑
Create GitHub Account & SSH Key Authentication
Day 1 · ~45 mins · github.com
0 / 8 ▼
Best practice: Always use SSH keys for GitHub authentication — never save your password in the terminal. SSH keys are more secure and don't expire.
01Create a free account at github.com using your institutional email
02Enable 2-Factor Authentication (2FA) under Settings → Password and Authentication
03Generate a new SSH key: ssh-keygen -t ed25519 -C "you@up.edu.ph"
04Start SSH agent and add key: eval "$(ssh-agent -s)" && ssh-add ~/.ssh/id_ed25519
05Copy your public key: cat ~/.ssh/id_ed25519.pub — copy the entire output
06Add SSH key to GitHub: Settings → SSH and GPG Keys → New SSH Key → paste key
07Test connection: ssh -T git@github.com — expected: "Hi username! You've successfully authenticated"
08Set up a GitHub profile: add a photo, bio, and your school/organization

Create Repository & Push Week 1 HTML
Day 1–2 · ~1 hour · First real commit!
0 / 9 ▼
Best practice: Always add a README.md, .gitignore, and a LICENSE to every repository. Never commit sensitive files like passwords, SSH keys, or .env files.
01Create a new public repository on GitHub named pgcv-hpc-internship
02Initialize with a README.md and select .gitignore template: Node
03Clone the repository to your local machine: git clone git@github.com:username/pgcv-hpc-internship.git
04Create a week1/ folder inside the repo and copy your week1_hpc_internship.html into it
05Stage the file: git add week1/week1_hpc_internship.html
06Commit with a proper message: git commit -m "feat: add week1 HPC internship lab checklist"
07Push to GitHub: git push origin main
08Verify the file appears on your GitHub repository page
09Update README.md: describe the project, list tech used, add your name — commit and push

Remote SSH into HPC Server via VS Code
Day 3 · ~45 mins · Edit server files from your laptop
0 / 6 ▼
This is a powerful skill — you can edit files on the remote HPC server as if they were local, with full VS Code features like autocomplete, linting, and Git integration.
01Open Command Palette: Ctrl+Shift+P → type Remote-SSH: Connect to Host
02Add new SSH host: enter ssh username@server-ip and save to ~/.ssh/config
03Connect and wait for VS Code Server to install on the remote machine
04Open your home directory on the server: File → Open Folder → /home/username
05Edit the Week 1 HTML file directly on the server — save and verify changes
06Open the integrated terminal: Ctrl+` — you now have a full shell on the HPC server

Git Workflow Inside VS Code
Day 4 · ~1 hour · Source Control panel
0 / 7 ▼
Best practice: Write meaningful commit messages using Conventional Commits format — feat:, fix:, docs:, chore: — so the history is readable.
01Open your cloned repository in VS Code: File → Open Folder → select pgcv-hpc-internship
02Click the Source Control icon (branch icon) in the left sidebar — explore the panel
03Make a change to README.md — notice the file appears under Changes in Source Control
04Stage the file by clicking the + icon — or stage all with Stage All Changes
05Write a commit message in the input box and press Ctrl+Enter to commit
06Push to GitHub using the sync button in the status bar or Source Control → Push
07Use GitLens to view the commit history: click any line → see who wrote it and when
# Conventional commit message format (best practice)
feat: add new feature
fix: correct a bug
docs: update README or comments
chore: update packages or config
refactor: restructure code without changing behavior
style: formatting changes only
